1 How it works

One Mac = one Apple ID = one iMessage line connected to Pairfon.

The agent is a small Node program that runs on the Mac and dials out to Pairfon Core — so the Mac needs no public IP, no port forwarding, no ngrok of its own. It:

  • Enrolls once with a pairing token → stores its own agent key.
  • Reads new incoming messages from the Messages database (chat.db) and pushes them to Pairfon.
  • Sends outgoing messages by driving Messages.app via AppleScript.
  • Reports delivered / read receipts and a heartbeat.
Because it reads your Messages database and controls Messages.app, macOS requires a few one-time privacy permissions. That's what Section 3 covers.

3 Permissions required

Three macOS privacy permissions. Read this — it explains which prompt automatically and which you must grant by hand.

PermissionWhy the agent needs itHow it's granted
Full Disk Access
Manual only		 To read the Messages database ~/Library/Messages/chat.db (incoming messages + receipts). Never prompts. You must add node by hand in System Settings — see Section 5.
Automation
(Apple Events)
Prompts		 To send messages by telling Messages.app / System Events what to type. A dialog appears the first time the agent sends — click OK / Allow.
Accessibility
Manual		 To let AppleScript control the keyboard for sending (the keystroke step). System Settings → Privacy & Security → Accessibility → add node (and Terminal).
Why doesn't Full Disk Access just pop a prompt?
Because chat.db is your entire private message history, Apple deliberately put it behind Full Disk Access — the one permission category that has no prompt and no API. No app (not ours, not even Apple's own) can request it; a human must toggle it in System Settings. This is by design, and it's a one-time step per Mac. Automation & Accessibility do prompt; Full Disk Access does not.
Critical: grant Full Disk Access & Accessibility to your node binary — not just Terminal. The agent runs under launchd, which launches node directly, so granting Terminal alone is not enough.

4 Install

Two ways. Method A (the app) is recommended — no Terminal at all.

Method A — Double-click installer app recommended

A real macOS app that runs every step for you with native dialogs (installs Node if needed, downloads the agent, enrolls, sets up auto-start, opens the permission pane).

1

Download it

Download Pairfon Installer.zip  →  double-click the zip to unpack Pairfon Installer.app.

2

Open it (first time only)

Because the app isn't code-signed yet, a plain double-click says "unidentified developer." Instead: right-click the app → Open → Open. After that it opens normally.

3

Fill in the popups

It asks for Core URL, pairing token (imt_…), and Apple ID handle, then installs & enrolls automatically. If Node isn't installed it adds it (one admin-password prompt); if Node is already there it skips that.

4

Grant Full Disk Access when it asks

At the end the app opens the Full Disk Access pane and waits — add node (see Section 5), and the app auto-detects it and finishes. Done.

The agent is bundled inside the app — it installs into ~/Library/Application Support/Pairfon/agent and sets up the auto-start service. No repo is cloned, so the source stays private. Logs: ~/Library/Application Support/Pairfon/install.log.

Method B — Terminal (advanced, team only)

Prefer the command line, or scripting many Macs? One command installs the tooling and the agent. Requires access to the private repo (internal/team use) — customers should use Method A.

Don't clone the repo inside the repo. Clone it to your home folder (~). A second copy inside imessage-api/imessage creates a nested path that breaks the auto-start service.
# from your HOME folder — installs Xcode CLT + Homebrew + Node, then the agent
git clone https://github.com/saif-ullah-k/core-3-in-1.git ~/core-3-in-1
cd ~/core-3-in-1/imessage-api/imessage
bash setup-mac.sh
Already have Node 18+ and Xcode CLT? Skip the bootstrap and run just bash install-agent.sh — setup-mac.sh simply calls it after preparing the tooling.

It asks three things, then sets everything up:

PromptWhat to enter
Core URLYour Pairfon Core address, e.g. https://api.pairfon.com
Pairing tokenThe imt_… token from Pairfon Ops
Apple ID handleThe email/phone shown in Messages, e.g. line@icloud.com

Both methods then:

  • enroll with Core (save config — no token/env ever needed again),
  • install a launchd service that auto-starts on login/reboot, auto-restarts if it stops, and keeps the Mac awake.

5 Grant Full Disk Access to node

The one manual permission. Do this once. The installer app opens this pane for you and auto-detects when you finish — these are the click-by-click steps.

1

Open the right settings pane

Apple menu → System Settings → Privacy & Security → Full Disk Access.

2

Click the + button

A file picker opens. node lives in a hidden folder, so you'll type the path — don't try to browse to it.

3

Jump to the binary

Press ⌘ + ⇧ Shift + G, then type the full path including node and press Enter:

/opt/homebrew/bin/node

Apple Silicon path shown above. On Intel Macs it's /usr/local/bin/node. Not sure? Run node -e "console.log(process.execPath)" on the Mac and use that path.

4

Select & toggle ON

The picker selects node (it may resolve to the real …/Cellar/node/<version>/bin/node — that's correct). Click Open, then make sure the toggle next to node is ON.

5

Restart the agent so it picks up the permission

macOS only applies a new permission when the process restarts:

launchctl unload ~/Library/LaunchAgents/com.pairfon.imessage-agent.plist
launchctl load   ~/Library/LaunchAgents/com.pairfon.imessage-agent.plist
While you're in Privacy & Security, also add node under Accessibility (same + → same path) so sending works. Tip: with the installer app you don't run the launchctl commands in step 5 — the app restarts the agent for you once it detects the toggle.

6 Verify it's working

Check the log on the Mac (the agent's location depends on how you installed).

# Method A — installer app:
tail -20 ~/Library/Application\ Support/Pairfon/agent/agent.log
# Method B — Terminal:
tail -20 ~/core-3-in-1/imessage-api/imessage/agent.log

You want to see both lines:

Pairfon iMessage agent running → https://…  (poll 3000ms, machine …)
Messages DB opened — inbound/status tracking active.
If you see “Messages DB opened”, Full Disk Access is correct and the Mac is fully live. It will now appear online in Pairfon, receive incoming messages, and send outgoing ones.
If you still see “Messages DB not readable yet”: the Full Disk Access toggle for node isn't ON, or you didn't restart the agent (Section 5, step 5). Fix and re-check.

8 Troubleshooting

“unable to open database file”

Full Disk Access isn't granted to node (or Messages isn't signed in yet). Do Section 5, confirm the toggle is ON, restart the agent.

Service shows exit status 1 / crash loop (launchctl list | grep pairfon shows - 1)

Usually a nested clone: the repo was cloned inside itself, so the launchd plist points at a doubled path like …/imessage/core-3-in-1/imessage-api/imessage/…. Fix:

# from the correct top-level checkout
cd ~/core-3-in-1/imessage-api/imessage
launchctl unload ~/Library/LaunchAgents/com.pairfon.imessage-agent.plist 2>/dev/null
rm -f ~/Library/LaunchAgents/com.pairfon.imessage-agent.plist
rm -rf ./core-3-in-1            # remove the nested copy
bash install-agent.sh          # re-run from the right place

“No agent key stored and no PAIRING_TOKEN set”

The agent never finished enrolling (often the nested-clone issue above). Re-run install-agent.sh from the correct folder with a fresh pairing token.

Messages send but nothing is received (or vice-versa)

Receiving needs Full Disk Access; sending needs Automation + Accessibility. If only one direction works, the missing permission is the one for the broken direction.

Agent shows offline in Pairfon after a reboot

The Mac rebooted to the login screen and nobody signed in. Enable automatic login so launchd can start the agent.
